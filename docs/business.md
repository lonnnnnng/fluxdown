# 业务文档

## 产品定位

FluxDown 是一款跨平台、多协议下载器。它不做资源发现和内容分发，而是专注于把用户已经拥有的下载源稳定地落地到本机设备。

核心定位：

- 面向个人用户和技术用户的通用下载工具。
- 桌面端同时覆盖命令行自动化和 GUI 日常使用。
- 移动端覆盖随手添加、查看和执行任务。
- 使用本地队列和本地文件系统，降低对服务端的依赖。

## 目标用户与场景

| 用户 | 场景 | FluxDown 提供的价值 |
| --- | --- | --- |
| 普通用户 | 下载网页直链、FTP 文件、m3u8 视频资源 | 一个 App/GUI 覆盖多个协议，减少切换工具。 |
| 技术用户 | 在脚本、服务器或开发机上批量下载 | CLI 提供 JSON 输出和队列控制，便于自动化。 |
| 跨平台用户 | 在 Windows、macOS、Linux、Android、iPhone 间使用相似体验 | 多端共享产品模型，降低学习成本。 |
| 发布维护者 | 为不同系统交付安装包和二进制 | GitHub Actions 统一构建和发布多端产物。 |

## 价值主张

- 多协议统一入口：把直链、P2P、HLS、文件服务器和特殊链接放到同一任务模型中。
- 多端覆盖：桌面端有 CLI 和 GUI，移动端有 App。
- 本地可控：队列、输出目录和下载文件都保存在用户设备上。
- 可审计发布：构建脚本、CI 作业和 Release manifest 可检查产物存在性和哈希。
- 可渐进增强：核心协议检测和任务模型独立于 UI，后续可以扩展同步、远程控制或更多后端。

## 商业边界

FluxDown 当前代码库没有内置收费、账号、广告或云服务模块。可选商业化方向应避免改变本地优先的基本体验：

- 专业版能力：远程队列同步、跨设备任务迁移、高级调度、自动分类。
- 企业/团队能力：私有协议后端、集中策略、审计日志、内网文件源适配。
- 支持服务：签名发布、商店上架、协议兼容性维护、定制集成。

任何商业化都需要额外设计隐私政策、服务条款、账号安全和支付合规；当前仓库尚未包含这些模块。

## 发布渠道

| 平台 | 渠道 | 当前状态 |
| --- | --- | --- |
| CLI | GitHub Release 二进制、源码构建 | CI 可生成 Linux、Windows、macOS CLI artifact。 |
| 桌面 GUI | GitHub Release 安装包/应用包 | CI 可生成 Linux `.deb`/`.rpm`、Windows 安装器、macOS `.app`/DMG。 |

## 合规与风险

- 下载工具可能被用于下载受版权保护的内容。FluxDown 只处理用户提供的链接，不提供内容搜索、索引或推荐。
- ed2k、BitTorrent 和 Magnet 属于 P2P 相关场景，发布文案和商店说明需要强调合法来源和用户责任。
- 移动端 torrent 依赖 `libtorrent_flutter`，其原生组件带有 GPL 许可证义务；正式分发前需要完成许可证审查。
- Apple App Store 和 Google Play 对下载器、P2P、后台下载、用户生成内容访问等场景可能有审核要求，需要在上架前确认政策。
- iOS 后台长期下载、外部 App 移交和文件系统访问受系统限制，不能按桌面端能力直接承诺。

## 指标建议

当前产品没有埋点和服务端。若后续引入遥测，建议只采集匿名、最小化指标，并默认给用户明确控制：

- 构建和发布指标：CI 成功率、产物体积、发布耗时。
- 客户端质量指标：崩溃率、下载失败类型、协议支持失败原因。
- 功能使用指标：协议类型分布、队列任务数量、暂停/继续使用率。

不要采集完整下载 URL、文件名、认证信息或本地路径，除非用户显式导出诊断信息。

## 业务决策记录

- 产品先做本地下载器，不引入服务端账号体系。
- 桌面端保留 CLI，一方面服务自动化，另一方面为 GUI 和 CI 提供可验证基础能力。
- ed2k 当前采用移交策略，避免在早期实现和维护完整 ed2k 协议栈。
- iOS IPA 和 Android 正式签名都通过外部密钥配置，不把签名材料放入仓库。
